+ The =VH-109=
  - already does wireless bridging to a LAN, so having the =FT= devices do that is
    super confusing and potetntially buggy.
  - Also a DHCP pool from =.200-.219= that can't conflict.
+ The =FT= devices will instead have their wifi connect to a =WAN=
  - and will provide LAN only and will not provide DHCP.
  - It is incredibly frustrating that almost ZERO laptop manufacturers provide
    LAN ports, so dongle' (_and_ transient ethernet devices) are necessary to deal
    with because consumers cannot be bothered to demand reasonable device specs.
  - Scripting network on devices with transient devices is a PITA. It's
    physically stateful and requires an extra layer or two.

** Config Backups

*** Config

FreshTomato provides =*.cfg= backups, but they're encoded (and I'm not sure how).
These serve as checkpoints that can easily be restored without scripting, but
they can easily be diffed. Each one is about 38KB.

*** NVRAM

** Usability Issues

*** Interface config

+ with =FT=, the =WAN= interface:
  - can't be placed under the same bridge =br0= and is handled differently --
    usually using a =gateway= abstraction for most router OS's. So it autocreates
    and autodestroys routes for other interfaces/bridges/vlans.
  - If there's a webadmin, then handling this WAN port differently would cause
    too many problems for noobs. So, to make these WebAdmins idiot proof -- you
    can't haz this port do whatever you want. This is nice because manual
    gateway configuration is very problematic


*** Connectivity

+ SSH configuration changes
  + non-interactive password auth isn't available for all OS and setting host
    config for static or dhcp ip's requires constantly changing this file (or
    some other host)
+ =systemd-newtorkd= requires changing the profile with =sudo= & symlinks or =ip
  addr add/del= commands, which =networkctl= doesn't seem to wipe out.

** SSH

*** Using =sshpass= (before keys are set on machine)

From [[https://superuser.com/questions/1605215/how-to-specify-password-in-ssh-command][s/o post]]. I'm unsure of how useful this is for Windows users.

#+begin_src shell
sshpass -ffilename ssh user@ip   # prefer this
sshpass -pPa5sw0rd ssh user@ip
#+end_src

So... this will non-interactively dump the NVRAM, which still needs its lines sorted

#+begin_src 
host=ft1
backup=$backupdir/freshtomato/$host-$(date +%s).nvram
sshpass -padmin ssh root@192.168.1.1 'nvram show' > $backup
#+end_src



*** From =ssh= via default password

You don't necessarily need SSH keys to make an SSH connection, so each NVRAM
reset requires clearing the relevant line from =~/.ssh/known_hosts=. But SSH does
not easily permit password entry for automation, so this cannot be totally
scripted unless =sshpass= is installed.

In any case, you'll need to change this code as the passwords/keys/networks
chnage.  requirements change

#+begin_src sshconfig
# host defaults
Host *
  IdentitiesOnly Yes
  UseRoaming no
  KexAlgorithms sntrup761x25519-sha512,sntrup761x25519-sha512@openssh.com,curve25519-sha256,curve25519-sha256@libssh.org,diffie-hellman-group-exchange-sha256

# or: Host myhost
Host 192.168.1.1
  HostName 192.168.1.1
  IdentitiesOnly no
  User root
  PasswordAuthentication yes
#+end_src

The authentication failure message will indicate line =n=. So run:

+ =vim ~/.ssh/known_hosts=
+ hit =28G= then =dd= to delete and =:wq= to save

.. if the line =n= were =28=. This will happen everytime you reset the router's
NVRAM to defaults bc it generates new host keys in =/etc/ssh=
  
Once you can authenticate without a =known_hosts= failure, then you can run
scripts against the box.

* Startup

If your box doesn't already have =FreshTomato=, you're on you own. That requires
flashing an intermediate image that lets you migrate away from the original
manufacturer image. This sometimes differs per-version of manufacturer software.
GLHF

Transitioning between custom firmwares usually doesn't require intermediate
flashing, but could be problematic. This is because the upgrade process for
custom firmwares is somewhat opaque and forward/backward migrations are NOT
necessarily restoring a known "zero" state for firmware upgrades. It depends,
but router hardware/arch varies considerably and so different hardware has
different stores for values. They usually depend on NVRAM for storage and FLASH
for NVRAM storage. But IIRC, some of NVRAM is dedicated to storing some of the
initial values (defaults).

** Upgrade

+ Find the upgrade file. It's a =zip=, that unpacks a =trx= file
+ Upgrade
  - Reset the box. On a Netgear R7000: hold reset for 8 seconds; the lights
    flash pink when you release the button. It reboots with NVRAM reinitialized.
  - Find =Administration > Upgrade=, submit the =trx= file, check that you want to
    =clear NVRAM= and Click =upgrade=

Upgrade, then (for me), login with =root/admin=.

** Capture initial NVRAM

Initially all ports will belong to a default network & bridge with =vlan 0=. This
is obnoxious if:

+ you want to retain access to the =FT1= default network
+ *but* you need the LAN ports to be on another network
+ *and* you can't control the VLANs in ethernet frames for the networks you're
  connected to (here, i'm connecting to a =VH-109= for FRC)

*** From Tools > System Commands

When you upgrade, you should clear NVRAM, unless you've upgraded very recently.
Capturing the NVRAM allows you to get insight into how the GUI changes as you
toggle settings.

_Immediately_ after loading the new firmware and without changing any value, open
=Tools > System Commands= and run:

#+begin_src shell
nvram show > /tmp/freshtomato-R7000-K26ARM-2025.5-AIO-64K.initial.nvram
#+end_src

It's difficult to manuevre in this shell, but this at least captures the fresh
values. Toggling certain settings will reboot the router, so you'll lose files
in =/tmp=

Keep in mind that this output contains passwords/keys and should be
GPG-encrypted. I don't really care for this application, so i'm not going to be
worried about that.

See [[id:2b08c8e0-8c78-4a29-a56d-16db0ef4be1e][DDWRT: Troubleshooting]] and [[id:94aaac5c-c45a-4b92-a977-d8d86cce3c2b][ddwrt: troubleshoot iptables]] for more info on
diffing these NVRAM updates. It's just important that you begin having captured
the "zeroed values" ... otherwise you can't diff and it won't be obvious which
=nvram show= invocations would contain your passwords in clear text.

** Change the bridge =br0= network addresses

Changing the =br0= configuration at once risks lockout, but is much faster than
migrating the LAN ports two at a time.

+ change IP space: move =br0= address to =10.100.4.247=.
  - This is all bits hot, except ~247 == (255-8)~
+ setup DHCP: change the pools from =.220= to =.239=. This should _not_ conflict with
  the =VH-109= DHCP, but should ultimately be removed. While configuring, this is
  convenient.
